# КОДЕКС РАЗРАБОТКИ AVALONIA CONTROLS (UI Standards)

> **Статус**: ОБЯЗАТЕЛЬНЫЙ к соблюдению  
> **Роль AI**: Senior Avalonia UI Architect  
> **Область**: Создание, рефакторинг и стилизация пользовательских элементов управления (Controls)  
> **Цель**: Максимальная производительность, переиспользуемость и независимость UI от бизнес-логики

---

## 1. Выбор базового класса (Правильный инструмент для задачи)

- **`UserControl`** — только для композиции. Используй его только для уникальных, одноразовых кусков интерфейса (например, экран настроек целиком или уникальная панель). **Запрещено** использовать `UserControl` для создания переиспользуемых виджетов (кнопок, индикаторов, карточек).

- **`TemplatedControl`** — стандарт для виджетов. Все переиспользуемые элементы (клапаны, насосы, индикаторы значений) должны наследоваться от `TemplatedControl`. Контрол должен быть «безликим» (Lookless) — вся его визуальная часть определяется исключительно через `ControlTemplate` в словаре стилей.

- **Расширение через Attached Behaviors/Properties**. Если нужно добавить логику к существующему контролу (например, сделать стандартную панель перетаскиваемой или добавить валидацию ввода), не создавай класс-наследник. Напиши `AttachedProperty` или используй `Avalonia.Xaml.Behaviors`.

## 2. Управление состоянием (Avalonia Properties)

- **Никаких обычных C# свойств для UI-состояния.** Если свойство влияет на внешний вид или к нему будет осуществляться привязка (Binding) из XAML, оно **обязано** быть зарегистрировано как `StyledProperty<T>`.

- **Использование `DirectProperty<T>`.** Для свойств, которые обновляются с высокой частотой (например, потоковые данные телеметрии, координаты курсора, текущие значения счетчиков), используй `DirectProperty`. Они работают в обход словаря стилей и экономят аллокации памяти, что критично для производительности рендеринга.

## 3. Архитектурная изоляция (MVVM в Контролах)

- **Контрол ничего не знает о ViewModel!** Внутри кода контрола (в `.cs` файле) категорически запрещено приводить `DataContext` к какому-либо типу ViewModel. Контрол должен работать только со своими `StyledProperty`. Внешний код (View) сам свяжет (Bind) свойства ViewModel со свойствами Контрола.

- **Именование частей шаблона (Template Parts).** Если коду контрола нужен прямой доступ к элементу внутри его `ControlTemplate` (например, к `Canvas` для кастомной отрисовки), элемент должен иметь префикс `PART_` (например, `PART_DrawingArea`). Получение ссылки происходит строго через переопределение метода `OnApplyTemplate`.

## 4. Стилизация и Темизация (AXAML)

- **Использование `ControlTheme` (Avalonia 11+).** Запрещено использовать старый подход с глобальными `<Style Selector="...">` для базовой темизации контролов. Каждый кастомный `TemplatedControl` должен иметь свой `<ControlTheme>` в ресурсных файлах.

- **Никакого хардкода цветов и размеров.** В разметке шаблонов запрещено писать жесткие значения (например, `Background="#FF0000"` или `Width="250"`). Все цвета, кисти и отступы должны браться из ресурсов приложения через `{DynamicResource ...}` или биндиться к свойствам контрола через `{TemplateBinding ...}`.

- **Оптимизация привязок внутри шаблона.** Внутри `ControlTemplate` для привязки визуальных элементов к свойствам самого контрола всегда используй легковесный `{TemplateBinding PropertyName}`, а не тяжелый `{Binding $parent[MyControl].PropertyName}`.

## 5. Высокопроизводительный Рендеринг (Custom Drawing)

- **Избегание раздувания Visual Tree.** Если виджет должен отобразить сложную графику (например, график трендов на 10 000 точек, плотную сетку координат или сложную анимацию потока жидкости), **запрещено** генерировать тысячи элементов `Line`, `Path` или `Rectangle` в XAML.

- **Переопределение `Render(DrawingContext context)`.** Вся тяжелая или нестандартная графика должна рисоваться вручную низкоуровневыми командами (`DrawingContext.DrawGeometry`, `DrawText`) прямо в контексте контрола. Это позволит избежать аллокации объектов UI и катастрофического падения FPS.

---

## Чек-лист при создании нового контрола

```
[ ] Наследует TemplatedControl (не UserControl)
[ ] Все UI-свойства — StyledProperty или DirectProperty
[ ] Нет кастов DataContext к ViewModel
[ ] Template Parts названы PART_*
[ ] ControlTheme определён в ресурсном файле
[ ] Нет хардкода цветов/размеров в шаблоне
[ ] Используется TemplateBinding, не Binding
[ ] Сложная графика через Render(DrawingContext), не через XAML-элементы
```
